# Архитектура продукта

## 1. Иерархия и роли в системе

Дополнительные правила регистрации и доступа:

- [Правила регистрации компаний и менеджеров](./company-registration-rules.md)

Порядок старшинства и сущностей:

1. создатель платформы
2. компания
3. руководитель компании
4. сотрудники компании
5. клиенты

Важно:

- `компания` это отдельная бизнес-сущность и отдельный контур данных
- `руководитель компании` это не сама компания, а первый пользователь внутри компании
- техническая роль руководителя компании в системе: `company_admin`

### Создатель платформы

Это верхний системный уровень.

Ему доступны:

- ручное создание управляющих компаний
- привязка компании к Telegram
- создание первого руководителя компании
- блокировка компании

Он не участвует в повседневной работе компании, а только открывает ей контур в системе.

### Компания

Это управляющая компания как отдельная организация в системе.

У компании есть:

- собственная карточка
- собственный Telegram-вход
- собственный контур данных
- собственные сотрудники
- собственные объекты, клиенты, платежи и документы

Компания идет по уровню доступа сразу после создателя платформы.

### Руководитель компании

Это первый пользователь компании.

Он работает от имени своей компании и открывает внутренний доступ для команды.

Что ему доступно:

- вход в кабинет компании
- просмотр данных только своей компании
- создание сотрудников своей компании
- назначение ролей внутри своей компании
- блокировка сотрудников своей компании

### Сотрудник управляющей компании

Работает через кабинет компании.

Что ему доступно:

- список объектов
- карточки клиентов
- заявки и задачи
- платежи
- договоры и документы
- отправка сообщений клиентам

### Клиент

Это собственник, арендатор или представитель клиента. Он работает через:

- Telegram-бот
- личный кабинет

Что ему доступно:

- мои объекты
- мои платежи
- мои заявки
- мои документы
- уведомления

## 2. Каналы работы

### Telegram-бот

Подходит для быстрых сценариев:

- старт и идентификация клиента или компании
- просмотр краткой информации
- создание заявки
- получение уведомлений
- переход в веб-кабинет по deep link

### Кабинет клиента

Подходит для расширенных сценариев:

- история платежей
- документы
- детали объекта
- просмотр статусов заявок
- профиль клиента

### Кабинет компании

Основной рабочий интерфейс команды:

- CRM по клиентам
- реестр объектов
- работа с заявками
- учет платежей
- журнал действий
- управление сотрудниками компании

## 3. Логическая схема

```text
Telegram Bot ----\
                  \
Client Cabinet -----> Backend API -----> PostgreSQL
                  /
Company Cabinet --/
```

## 4. Почему нужен backend между интерфейсами и базой данных

Если подключать Telegram и кабинеты напрямую к базе, быстро появятся проблемы:

- дублирование логики
- разные форматы данных
- сложная авторизация
- конфликтующие записи
- трудно добавить валидацию и аудит

Поэтому все интерфейсы должны работать только с backend API.

## 5. Базовые модули backend

### Auth module

- вход по Telegram
- проверка роли
- привязка Telegram user к клиенту или компании

### Companies module

- карточка компании
- Telegram-привязка компании
- временный пароль компании
- смена пароля при первом входе

### Staff module

- руководитель компании
- менеджеры, поддержка, бухгалтерия
- роли и статусы сотрудников

### Clients module

- карточка клиента
- тип клиента: owner, tenant, partner
- связь с объектами

### Properties module

- объекты недвижимости
- адрес
- статус
- тип объекта
- управляющий менеджер

### Requests module

- заявки клиентов
- тип заявки
- приоритет
- исполнитель
- статус

### Payments module

- начисления
- оплаты
- задолженности
- период

### Documents module

- договоры
- акты
- квитанции
- ссылки на Google Drive

### Notifications module

- сообщения в Telegram
- шаблоны
- события уведомлений

## 6. Рекомендуемый MVP

### Для клиента

- авторизация через Telegram
- просмотр своих объектов
- просмотр платежей
- создание заявки
- просмотр статуса заявки

### Для компании

- вход через Telegram и временный пароль
- обязательная смена пароля при первом входе
- кабинет руководителя компании
- список сотрудников компании
- список объектов и клиентов компании

### Для системы

- хранение данных в PostgreSQL
- уведомления через Telegram
- журнал действий

## 7. Масштабирование

Архитектура уже сразу строится на PostgreSQL, поэтому масштабирование дальше идет через:

- новые модули backend
- миграции схемы
- очереди и фоновые задачи
- репликацию и резервное копирование
