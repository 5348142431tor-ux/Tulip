# Code Structure

## Цель

Этот документ фиксирует, как развивать `Tulip` последовательно и без хаоса.

Главный принцип:

- структура проекта должна быть понятной новому разработчику
- каждый файл должен иметь одну ясную ответственность
- код должен раскладываться по слоям, а не смешиваться в одном месте

## 1. Верхний уровень проекта

```text
apps/         Приложения проекта
packages/     Общие модули и доменные части
docs/         Документация и правила проекта
infra/        База данных, миграции, инфраструктура
deploy/       Production-деплой и публичная конфигурация
prototype/    Быстрые визуальные прототипы
```

## 2. Правило разделения

Мы не используем подход:

- одна функция = один файл

Мы используем подход:

- одна ответственность = один модуль

То есть файл создается не потому, что функция новая, а потому что появляется отдельная зона ответственности.

## 3. Структура backend

Целевая структура `apps/api`:

```text
apps/api/
  src/
    routes/         HTTP endpoints
    services/       бизнес-логика и сценарии
    repositories/   работа с БД
    sql/            SQL-запросы
    auth/           пароли, access codes, auth helpers
    lib/            утилиты
    config/         конфиг и env
    server.js       точка входа
    db.js           подключение к PostgreSQL
```

### Роли слоев

`routes/`

- принимают запрос
- валидируют обязательные поля на базовом уровне
- вызывают service или repository
- формируют HTTP-ответ

`services/`

- содержат бизнес-правила
- координируют несколько репозиториев
- не должны знать про HTML или Telegram UI

`repositories/`

- работают с PostgreSQL
- получают и сохраняют данные
- не должны содержать лишнюю HTTP-логику

`sql/`

- хранят SQL отдельно от бизнес-логики
- один набор SQL на одну сущность или сценарий

`auth/`

- парольные функции
- access links
- проверка auth-сценариев

## 4. Структура frontend

Для каждого кабинета целевая структура такая:

```text
apps/<cabinet>/
  index.html
  styles.css
  app.js               временная точка входа для MVP
  src/
    pages/             экраны
    components/        переиспользуемые UI-блоки
    api/               вызовы backend
    state/             local/session state
    utils/             форматирование и helpers
    styles/            общие стили
```

Пока MVP может жить на `index.html + app.js + styles.css`, но как только файл становится слишком большим, мы режем его по экранным модулям.

## 5. Когда нужно делить файл

Файл пора делить, если:

- в нем больше одной большой ответственности
- его трудно описать одной фразой
- в нем смешаны UI, API, storage и бизнес-правила
- при изменении одной части легко сломать другую

Пример:

- если `admin-cabinet/app.js` содержит dashboard, properties, companies, payments и auth, его нужно делить по модулям

## 6. Правила именования

Используем последовательные имена:

- `companyRoutes`
- `companyService`
- `companyRepository`
- `companySql`

Для файлов:

- `companyAuthRepository.js`
- `companies.js` в `sql/`
- `companyAuth.js` в `sql/`

Не используем случайные имена и не смешиваем разные стили именования в одном слое.

## 7. Правила роста проекта

Каждая новая фича проходит так:

1. определить сущность или сценарий
2. понять, в какой app это относится
3. разложить по слоям
4. только потом писать код

### Пример

Если добавляем вход компании:

- route: login endpoint
- auth/service: проверка пароля
- repository: чтение компании и access link
- sql: запросы к PostgreSQL
- bot: отдельная ветка меню
- company-cabinet: отдельный экран входа

## 8. Правила для БД

- новые изменения схемы только через `infra/postgres/migrations`
- старые рабочие данные не ломаем
- seed не используем как место для новых боевых изменений
- если логика уже используется, добавляем новую миграцию, а не переписываем историю

## 9. Правила для документации

Документация добавляется, если:

- появляется новая архитектурная роль
- появляется новый auth-flow
- появляется новая инфраструктурная схема
- появляется правило, которое команда должна соблюдать постоянно

## 10. Что делать прямо сейчас в Tulip

Текущее состояние:

- backend уже частично разложен правильно
- `admin-cabinet/app.js` перегружен и должен быть следующим кандидатом на разрезание
- auth компании уже выделен как отдельная зона ответственности

Следующие правильные шаги:

1. разделить `admin-cabinet` на модули по экранам
2. постепенно ввести `services/` в `apps/api`
3. держать все новые auth-сценарии отдельно от property/payment логики

## 11. Главный стандарт проекта

Новый код должен быть:

- понятен
- локализован по ответственности
- предсказуем по имени файла
- расширяем без переписывания половины проекта

Если есть сомнение, как положить новый код, выбираем структуру, которую проще всего объяснить новому разработчику за 2 минуты.
