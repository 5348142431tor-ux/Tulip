# Database Architecture

Основное хранилище проекта теперь рекомендуется строить на `PostgreSQL`.

## Почему PostgreSQL как основная база

Для ядра системы управляющей компании нужна связная БД:

- компании
- дома
- помещения
- собственники
- доли владения
- долги
- платежи
- заявки
- документы

Для этого `PostgreSQL` подходит намного лучше.

## Принцип хранения

Источник истины:

- `PostgreSQL`

Вспомогательные каналы:

- `Google Drive`
- `Telegram`

## Основные таблицы

### `management_companies`

Компания как отдельная сущность верхнего уровня после создателя платформы.

Она не равна сотруднику и не равна роли.

Для модели доступа здесь должны храниться:

- `code`
- `title`
- `telegram_id`
- `telegram_username`
- `temp_password`
- `password_hash`
- `must_change_password`
- `status`

Именно эта таблица открывает отдельный контур данных компании.

### `staff`

Сотрудники компании:

- руководитель компании
- менеджеры
- поддержка
- бухгалтерия

Для модели доступа здесь также должны храниться:

- `management_company_id`
- `role`
- `telegram_id`
- `telegram_username`
- `status`

Первый сотрудник компании создается владельцем платформы вручную с ролью `company_admin`.
Дальше уже он сам создает менеджеров и других сотрудников своей компании.

### `clients`

Клиенты и собственники.

### `properties`

Дом или объект под управлением.

### `units`

Помещения внутри дома.

### `ownerships`

Связь помещение -> собственник, включая долю.

### `unit_balances`

Текущий баланс помещения.

### `charges`

Начисления, задолженности и оплаты по периодам.

### `requests`

Заявки клиентов и внутренних сотрудников.

### `documents`

Документы по дому, помещению или клиенту.

### `activity_logs`

Журнал действий для аудита.

## Ключевой принцип безопасности данных

Мы не удаляем уже внесенные записи при доработках интерфейса или логики.

Рабочее правило:

- структура мигрируется мягко
- старые записи сохраняются
- новые поля добавляются совместимо
- изменения схемы делаются через новые миграции, а не через обнуление данных

## Как запускать локально

Из корня проекта:

```bash
docker compose up -d
```

После запуска:

- PostgreSQL: `localhost:5432`
- Adminer: `http://localhost:8080`

Параметры подключения:

- database: `tulip`
- user: `tulip`
- password: `tulip_dev_password`
